## 二、网关日志详解

### 不同错误类型的网关日志对比

#### ✅ 正常请求
```json
{
  "request_id": "a1b2c3",
  "timestamp": "2026-04-19T10:00:00Z",
  "method": "GET",
  "uri": "/api/v1/orders/123",
  "status": 200,
  "latency_ms": 35,
  "client_ip": "183.12.xx.xx",
  "upstream": "10.0.0.5:8080",
  "upstream_status": 200
}
```

#### ❌ 401 — Token 失效/未登录（网关拦截，请求根本没到后端）
```json
{
  "request_id": "b2c3d4",
  "timestamp": "2026-04-19T10:01:00Z",
  "method": "POST",
  "uri": "/api/v1/orders",
  "status": 401,
  "latency_ms": 2,          // ← 极短，网关直接返回，没转发
  "client_ip": "183.12.xx.xx",
  "upstream": null,          // ← 没有后端地址，证明请求被网关拦截了
  "error": "JWT token expired"
}
```

#### ❌ 429 — 限流（网关拦截）
```json
{
  "status": 429,
  "latency_ms": 1,
  "upstream": null,
  "error": "rate limit exceeded, limit: 100/min, current: 101"
}
```

#### ❌ 502 — 后端服务挂了（网关转发失败）
```json
{
  "request_id": "c3d4e5",
  "status": 502,
  "latency_ms": 5001,        // ← 等了5秒超时
  "upstream": "10.0.0.5:8080",
  "upstream_status": null,   // ← 后端没有响应
  "error": "upstream connect error: connection refused"
}
```

#### ❌ 504 — 后端超时（转发成功但后端太慢）
```json
{
  "status": 504,
  "latency_ms": 30000,       // ← 等了30秒
  "upstream": "10.0.0.5:8080",
  "upstream_status": null,
  "error": "upstream timeout after 30000ms"
}
```

#### ❌ 500 — 后端代码报错（网关正常转发，是后端的问题）
```json
{
  "status": 500,
  "latency_ms": 120,         // ← 正常转发了，后端执行出错
  "upstream": "10.0.0.5:8080",
  "upstream_status": 500,    // ← 后端返回了 500
  "error": null              // ← 网关本身没问题
}
```

---

### 实际场景举例

> **场景：用户反馈下单失败**

```
排查步骤：

1. 先看网关日志
   └── status=502，upstream=null，error="connection refused"
       → 结论：后端服务根本没启动或崩溃了，不是代码 bug
       → 去重启服务 / 看容器日志

   └── status=500，upstream_status=500，latency=120ms
       → 结论：网关正常，是后端代码报错
       → 去看应用日志找堆栈

   └── status=401
       → 结论：Token 问题，让前端检查 Token 是否过期
```

---

### Web 服务器日志 vs 网关日志的区别

| 对比维度 | 网关日志 | Web 服务器日志（Nginx 反向代理） |
|---|---|---|
| **定位** | 流量治理层：鉴权、限流、路由、负载均衡 | 静态资源服务 + HTTP 协议处理 |
| **典型产品** | Kong、APISIX、Spring Cloud Gateway | Nginx、Apache |
| **记录重点** | consumer（谁调的）、route（匹配哪条路由）、插件执行结果 | HTTP 协议细节、响应字节数、Referer |
| **有无鉴权信息** | ✅ 有（JWT、API Key 校验结果） | ❌ 没有 |
| **有无限流信息** | ✅ 有 | ❌ 没有 |
| **上游信息** | upstream 服务名（order-service） | upstream IP:端口 |
| **使用场景** | 微服务架构，多服务路由 | 传统单体架构或托管前端静态页面 |

> 简单理解：**Nginx 是网关的一种实现**，但功能比专业 API 网关简单很多。在微服务里，Nginx 可能只作为最外层的 7 层负载均衡，后面还有专业 API 网关。

## 三、`/var` 是服务器中的什么路径

`/var` 是 Linux 文件系统的**标准顶级目录**，全称 **variable**（可变数据）。

### Linux 根目录结构
```
/                   # 根目录
├── etc/            # 配置文件（nginx.conf 在这里）
├── var/            # 可变数据（会随程序运行不断变化）
│   ├── log/        # ← 最重要！所有日志在这里
│   │   ├── nginx/
│   │   │   ├── access.log
│   │   │   └── error.log
│   │   ├── mysql/
│   │   │   ├── error.log
│   │   │   └── slow.log
│   │   ├── syslog      # 系统日志
│   │   └── auth.log    # 登录认证日志
│   ├── lib/        # 程序运行时数据（MySQL 数据库文件默认在 /var/lib/mysql）
│   ├── cache/      # 缓存文件
│   ├── run/        # PID 文件（记录进程 ID）
│   └── spool/      # 队列数据（邮件队列等）
├── usr/            # 用户程序（可执行文件、库）
├── home/           # 普通用户家目录
├── root/           # root 用户家目录
├── tmp/            # 临时文件
├── proc/           # 虚拟文件系统，反映内核状态
└── bin/            # 基础命令（ls、cp、mv）
```

### 实际工作中常用路径
```bash
/var/log/nginx/access.log   # Nginx 访问日志
/var/log/nginx/error.log    # Nginx 错误日志
/var/log/mysql/slow.log     # MySQL 慢查询日志
/var/log/mysql/error.log    # MySQL 错误日志
/var/log/syslog             # 系统日志
/var/lib/mysql/             # MySQL 数据文件
/var/lib/docker/            # Docker 容器数据
```

## 四、网关日志为什么只有客户端信息

这是个很好的问题，根本原因是**网关的职责是处理"进来的请求"**。

### 网关的位置
```
客户端（手机/浏览器）
        ↓  请求进来
   [ 网关 ]           ← 网关日志记录这里发生的事
        ↓  转发
   [ 后端服务 ]        ← 后端响应信息记录在应用日志里
        ↓  查数据库
   [ 数据库 ]
```

### 为什么网关日志"缺少"服务器端信息

| 信息类型 | 在哪里记录 | 原因 |
|---|---|---|
| 客户端 IP、请求方法、路径、Token | 网关日志 ✅ | 请求到达网关时就能拿到 |
| upstream IP + 响应状态码 | 网关日志 ✅ | 网关转发后能拿到 HTTP 状态 |
| 后端业务逻辑、变量值、异常堆栈 | 应用日志 ✅ | 网关不执行业务代码，看不到 |
| SQL 执行情况 | 数据库日志 ✅ | 网关根本不碰数据库 |
| 函数调用栈 | 应用日志 ✅ | 网关不知道后端代码怎么跑的 |

### 网关实际上能记录的服务端信息
```json
{
  "upstream": "10.0.0.5:8080",      // ← 转发到哪台后端
  "upstream_status": 500,            // ← 后端返回的 HTTP 状态码
  "upstream_latency_ms": 3200,       // ← 后端处理耗时
  "upstream_response_size": 1024     // ← 后端返回的响应大小
}
```

> 网关只能看到"后端返回了什么状态码和耗时"，但看不到后端里面发生了什么。这就是为什么排查 500 错误，看完网关日志还要去看**应用日志**。

## 五、工作中最常用的 SQL 语句

### 1. SELECT — 查询基础
```sql
-- 基础查询
SELECT * FROM orders;

-- 指定字段（实际工作中不要用 *，性能差）
SELECT id, user_id, amount, status, created_at FROM orders;

-- 条件过滤
SELECT * FROM orders WHERE status = 'pending' AND amount > 100;

-- 模糊查询
SELECT * FROM users WHERE name LIKE '张%';   -- 以张开头
SELECT * FROM users WHERE name LIKE '%张%';  -- 包含张

-- 范围查询
SELECT * FROM orders WHERE created_at BETWEEN '2026-01-01' AND '2026-04-19';
SELECT * FROM orders WHERE amount BETWEEN 100 AND 500;

-- IN 查询（多值匹配）
SELECT * FROM orders WHERE status IN ('pending', 'paid', 'shipped');

-- NULL 判断
SELECT * FROM users WHERE deleted_at IS NULL;     -- 未删除的
SELECT * FROM users WHERE avatar IS NOT NULL;     -- 有头像的
```

### 2. 排序 & 分页
```sql
-- 排序
SELECT * FROM orders ORDER BY created_at DESC;          -- 最新的在前
SELECT * FROM orders ORDER BY amount DESC, created_at ASC; -- 多字段排序

-- 分页（MySQL）
SELECT * FROM orders 
WHERE status = 'paid'
ORDER BY created_at DESC
LIMIT 20 OFFSET 0;    -- 第1页，每页20条

-- 第2页
LIMIT 20 OFFSET 20;
-- 计算公式：OFFSET = (page - 1) * size
```

### 3. 聚合统计 — 工作中最高频
```sql
-- 计数
SELECT COUNT(*) FROM orders;                          -- 总订单数
SELECT COUNT(*) FROM orders WHERE status = 'paid';    -- 已支付订单数
SELECT COUNT(DISTINCT user_id) FROM orders;           -- 下单用户数（去重）

-- 求和、平均、最大最小
SELECT 
  SUM(amount)     AS total_amount,    -- 总金额
  AVG(amount)     AS avg_amount,      -- 平均金额
  MAX(amount)     AS max_amount,      -- 最大金额
  MIN(amount)     AS min_amount       -- 最小金额
FROM orders 
WHERE status = 'paid';

-- 按维度分组统计（最常用！）
SELECT 
  status,
  COUNT(*)    AS order_count,
  SUM(amount) AS total_amount
FROM orders
GROUP BY status;

-- 结果示例：
-- pending  | 120 | 15000.00
-- paid     | 980 | 128000.00
-- refunded | 45  | 5600.00

-- 按天统计订单量（日报表）
SELECT 
  DATE(created_at) AS date,
  COUNT(*)         AS order_count,
  SUM(amount)      AS daily_amount
FROM orders
GROUP BY DATE(created_at)
ORDER BY date DESC;

-- HAVING（对分组结果过滤，和 WHERE 不同）
-- WHERE 过滤原始行，HAVING 过滤分组后的结果
SELECT user_id, COUNT(*) AS order_count
FROM orders
GROUP BY user_id
HAVING order_count > 5;   -- 找出下单超过5次的用户
```

### 4. JOIN 关联查询
```sql
-- INNER JOIN（只返回两表都有的数据）
SELECT 
  o.id         AS order_id,
  u.name       AS user_name,
  u.phone,
  o.amount,
  o.status
FROM orders o
INNER JOIN users u ON o.user_id = u.id
WHERE o.status = 'paid';

-- LEFT JOIN（保留左表所有数据，右表没有则为 NULL）
-- 场景：找出注册了但从未下过单的用户
SELECT 
  u.id,
  u.name,
  COUNT(o.id) AS order_count
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.name
HAVING order_count = 0;

-- 三表关联
SELECT 
  o.id       AS order_id,
  u.name     AS user_name,
  p.name     AS product_name,
  oi.qty,
  oi.price
FROM orders o
JOIN users u        ON o.user_id = u.id
JOIN order_items oi ON o.id = oi.order_id
JOIN products p     ON oi.product_id = p.id
WHERE o.id = 10086;
```

### 5. INSERT — 插入
```sql
-- 插入单条
INSERT INTO users (name, phone, email, created_at)
VALUES ('张三', '13800138000', 'zhangsan@example.com', NOW());

-- 插入多条
INSERT INTO users (name, phone) VALUES
  ('张三', '13800000001'),
  ('李四', '13800000002'),
  ('王五', '13800000003');

-- 查询结果插入另一张表（常用于数据归档）
INSERT INTO orders_archive
SELECT * FROM orders WHERE created_at < '2025-01-01';
```

### 6. UPDATE — 更新
```sql
-- 基础更新（注意一定要加 WHERE！）
UPDATE orders SET status = 'cancelled' WHERE id = 10086;

-- 多字段更新
UPDATE users 
SET name = '李四', updated_at = NOW() 
WHERE id = 123;

-- 批量更新（用 CASE WHEN）
UPDATE orders
SET status = CASE 
  WHEN id = 1 THEN 'paid'
  WHEN id = 2 THEN 'cancelled'
  WHEN id = 3 THEN 'shipped'
END
WHERE id IN (1, 2, 3);
```

### 7. DELETE — 删除
```sql
-- 删除单条（实际生产很少用物理删除！）
DELETE FROM orders WHERE id = 10086;

-- 实际工作中用软删除（推荐）
UPDATE orders SET deleted_at = NOW() WHERE id = 10086;

-- 清空测试数据（慎用）
DELETE FROM orders WHERE created_at > '2026-04-01' AND user_id = 9999;
```

### 8. 子查询
```sql
-- IN 子查询：找出有未支付订单的用户信息
SELECT * FROM users 
WHERE id IN (
  SELECT DISTINCT user_id FROM orders WHERE status = 'pending'
);

-- EXISTS 子查询（性能比 IN 好）
SELECT * FROM users u
WHERE EXISTS (
  SELECT 1 FROM orders o 
  WHERE o.user_id = u.id AND o.status = 'pending'
);
```

### 9. 实际工作常用组合模板
```sql
-- 业务报表模板（最常写的）
SELECT 
  DATE(o.created_at)    AS date,
  COUNT(o.id)           AS order_count,
  COUNT(DISTINCT o.user_id) AS user_count,
  SUM(o.amount)         AS gmv,
  AVG(o.amount)         AS avg_order_value
FROM orders o
WHERE 
  o.created_at >= '2026-04-01'
  AND o.created_at < '2026-04-20'
  AND o.status != 'cancelled'
  AND o.deleted_at IS NULL
GROUP BY DATE(o.created_at)
ORDER BY date DESC;

-- 排查某用户的完整订单记录
SELECT 
  o.id, o.status, o.amount, o.created_at,
  GROUP_CONCAT(p.name) AS products   -- 把多行合并成一行
FROM orders o
JOIN order_items oi ON o.id = oi.order_id
JOIN products p     ON oi.product_id = p.id
WHERE o.user_id = 10086
GROUP BY o.id
ORDER BY o.created_at DESC
LIMIT 10;
```

### 10. 常用内置函数速查
```sql
-- 时间函数
NOW()                          -- 当前时间：2026-04-19 10:23:45
DATE(created_at)               -- 只取日期：2026-04-19
DATE_FORMAT(created_at, '%Y-%m') -- 格式化：2026-04
DATEDIFF(NOW(), created_at)    -- 相差天数
DATE_SUB(NOW(), INTERVAL 7 DAY) -- 7天前

-- 字符串函数
CONCAT(first_name, ' ', last_name)  -- 拼接字符串
LENGTH(name)                         -- 字符串长度
UPPER(email) / LOWER(email)          -- 大小写转换
SUBSTRING(phone, 1, 3)              -- 截取：138
REPLACE(content, '敏感词', '***')   -- 替换

-- 数字函数
ROUND(amount, 2)    -- 保留2位小数
FLOOR(3.9)          -- 向下取整：3
CEIL(3.1)           -- 向上取整：4
ABS(-100)           -- 绝对值：100

-- 条件函数
IF(status = 'paid', '已支付', '未支付')
IFNULL(remark, '无备注')                          -- NULL 替换
COALESCE(phone, email, '未填写')                  -- 取第一个非 NULL
CASE WHEN amount > 1000 THEN '大单' ELSE '小单' END
```
