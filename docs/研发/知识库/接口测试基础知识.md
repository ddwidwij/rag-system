接口测试要学的“基础知识”，本质上可以分成 8 块：

**接口是什么、怎么通信、怎么设计用例、怎么执行、怎么定位问题、怎么做自动化、怎么管测试环境、怎么保证质量。**

下面我按“真正做接口测试必须会的东西”给你系统整理一版。

---

# 一、先搞懂：什么是接口测试

接口测试，通常就是测试系统与系统之间、前端与后端之间的数据交互是否正确。

你要验证的不只是“能不能返回数据”，还包括：

* 请求是否成功到达服务端
* 参数校验是否正确
* 返回结果是否符合预期
* 异常输入是否被正确处理
* 权限控制是否正确
* 数据是否真的写进数据库
* 多系统调用时流程是否正确

你可以把它理解成：

**不测页面外观，重点测业务逻辑和数据交互。**

# 二、必须会的接口基础概念

## 1. API / 接口

API 就是系统提供给别人调用的“功能入口”。

例如：

* 登录接口
* 查询用户信息接口
* 下单接口
* 支付接口

## 2. 请求和响应

一次接口调用，核心就是：

* 客户端发起 **请求**
* 服务端返回 **响应**

### 请求里常见内容

* 请求地址 URL
* 请求方法
* 请求头 Header
* 请求参数
* 请求体 Body

### 响应里常见内容

* 状态码
* 响应头
* 响应体
* 返回时间

## 3. URL

接口地址一般长这样：

```text
https://api.xxx.com/user/login
```

你至少要看懂：

* `https`：协议
* `api.xxx.com`：域名
* `/user/login`：路径

## 4. HTTP / HTTPS

接口测试最基本的通信协议就是 HTTP/HTTPS。

你要掌握：

* HTTP 是怎么传输请求和响应的
* HTTPS 比 HTTP 多了一层加密
* 大多数线上接口都走 HTTPS

# 三、HTTP 基础：这是接口测试的核心地基

## 1. 常见请求方法

### GET

一般用于查询数据。

例如：

* 查询用户信息
* 查询订单列表

### POST

一般用于新增、提交数据。

例如：

* 登录
* 创建订单

### PUT

一般用于整体更新数据。

### PATCH

一般用于部分更新数据。

### DELETE

一般用于删除数据。

你至少要知道：

**不同方法代表不同语义，不能乱用。**

## 2. 常见状态码

这个必须背熟。

### 2xx 成功

* `200 OK`：请求成功
* `201 Created`：创建成功
* `204 No Content`：成功但无返回体

### 3xx 重定向

接口测试里相对少见，但要知道存在。

### 4xx 客户端错误

* `400 Bad Request`：请求参数有问题
* `401 Unauthorized`：未登录/认证失败
* `403 Forbidden`：无权限
* `404 Not Found`：资源不存在
* `405 Method Not Allowed`：请求方法不支持
* `409 Conflict`：资源冲突
* `415 Unsupported Media Type`：请求格式不支持
* `422 Unprocessable Entity`：参数格式合法但业务校验不通过

### 5xx 服务端错误

* `500 Internal Server Error`：服务内部异常
* `502 Bad Gateway`
* `503 Service Unavailable`
* `504 Gateway Timeout`

你做接口测试时，不能只看 `200`，要判断：

**状态码是否符合接口设计。**

## 3. Header

常见请求头：

* `Content-Type`
* `Authorization`
* `Cookie`
* `User-Agent`
* `Accept`

最重要的是：

### `Content-Type`

表示请求体是什么格式，比如：

* `application/json`
* `application/x-www-form-urlencoded`
* `multipart/form-data`

# 四、参数知识：接口测试天天都在用

## 1. 参数传递位置

你要分清参数放在哪里：

### 路径参数

```text
GET /users/1001
```

这里 `1001` 就是路径参数。

### 查询参数 Query

```text
GET /users?page=1&size=10
```

### 请求头参数 Header

比如 token。

### 请求体参数 Body

一般 POST/PUT/PATCH 常见。

## 2. 参数类型

要理解常见数据类型：

* string
* int
* float
* boolean
* array
* object
* null

## 3. 参数校验

测试时必须覆盖：

* 必填项为空
* 参数类型错误
* 参数长度超限
* 参数范围越界
* 枚举值非法
* 重复提交
* 特殊字符
* 中文/空格/emoji
* 时间格式错误

---

# 五、数据格式：重点掌握 JSON

现在大多数接口都用 JSON。

例如：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "userId": 1001,
    "name": "Tom"
  }
}
```

你必须会：

* 读 JSON
* 写 JSON
* 校验 JSON 字段
* 判断字段类型
* 判断字段是否缺失
* 判断字段值是否正确

还要理解：

* 对象 object
* 数组 array
* 嵌套结构
* 空值 null
* 布尔值 true/false

除了 JSON，还要知道：

* form 表单
* file 上传
* XML（老系统可能有）

# 六、认证与鉴权：很多接口问题都卡在这里

## 1. 认证 Authentication

证明“你是谁”。

常见方式：

* 用户名 + 密码
* Token
* JWT
* Session + Cookie
* AppKey / AppSecret
* OAuth 2.0

## 2. 鉴权 Authorization

判断“你有没有权限做这件事”。

例如：

* 普通用户不能删别人订单
* 未登录用户不能查个人信息
* 非管理员不能操作后台接口

接口测试里，认证和鉴权一定要测：

* 不带 token
* token 失效
* token 篡改
* 权限不足
* 越权访问

### 越权很重要

分两类：

* **水平越权**：A 用户访问 B 用户的数据
* **垂直越权**：普通用户访问管理员功能

---

# 七、Cookie、Session、Token 的区别

这个经常考，也经常在工作里碰到。

## 1. Cookie

保存在客户端的一小段数据。

## 2. Session

保存在服务端的会话信息。

## 3. Token

服务端签发给客户端的身份凭证，后续请求带上。

现在很多前后端分离项目更常见的是：

**登录成功后返回 token，后续请求通过 Header 携带 token。**

# 八、必须掌握的接口测试用例设计

这部分最关键。你会点 Postman 不等于会接口测试。

## 1. 正常场景

也叫主流程。

例如登录接口：

* 正确用户名 + 正确密码

## 2. 异常场景

这是测试价值所在。

例如：

* 用户名为空
* 密码为空
* 用户不存在
* 密码错误
* 参数超长
* 参数含特殊字符
* 缺少必填字段
* 重复请求

## 3. 边界值

例如：

* 最小长度
* 最大长度
* 刚好超出长度
* 最小数值
* 最大数值
* 日期边界

## 4. 等价类

把输入按“有效/无效”分类，用代表值减少用例数量。

## 5. 业务规则

例如下单接口：

* 库存不足不能下单
* 未登录不能下单
* 商品下架不能下单
* 金额计算必须正确

## 6. 状态流转

例如订单：

* 待支付
* 已支付
* 已发货
* 已完成
* 已取消

你要测：

**非法状态是否能被拦住。**

## 7. 幂等性

这个很重要。

例如支付回调、提交订单、退款接口，重复调用时是否会产生重复结果。

常见测试：

* 连续调用两次
* 网络超时后重试
* 同一请求重复提交

## 8. 关联场景

很多接口不是独立的。

例如：

1. 先登录
2. 拿 token
3. 创建订单
4. 支付订单
5. 查询订单状态

这种就是接口关联测试。

# 九、返回结果怎么校验

不能只看“有返回”。

至少要校验：

## 1. 状态码

是不是预期的 200、400、401 等。

## 2. 业务码

很多系统除了 HTTP 状态码，还有自己的业务码，例如：

```json
{
  "code": 10000,
  "msg": "success"
}
```

## 3. 返回字段

* 字段是否存在
* 类型是否正确
* 值是否正确
* 是否多返回敏感字段

## 4. 响应时间

太慢也算问题。

## 5. 数据库结果

例如新增用户成功后，要查库确认数据真的入库了。

# 十、接口测试必须懂一点数据库

不要求你一开始就很强，但基础 SQL 必须会。

至少会：

* `select`
* `insert`
* `update`
* `delete`
* 条件查询
* 排序
* 分组
* 联表查询基础

接口测试常见用途：

* 校验数据是否写入成功
* 校验字段是否更新正确
* 校验脏数据是否产生
* 校验事务是否回滚

例如你测注册接口，不能只看返回成功，还要查数据库：

* 用户表是否新增记录
* 默认状态是否正确
* 密码是否加密存储

# 十一、还要懂一点 Linux 和日志

很多接口问题不是接口本身，是环境、服务、日志。

你至少要懂：

* 看服务是否启动
* 看日志文件
* 基本命令行操作
* 查端口
* 查进程
* 看网络连通性

常见命令思路：

* 查看文件
* 搜索日志关键字
* 查看端口占用
* 查看进程
* 查看 CPU / 内存

接口报 500 时，你经常要结合日志判断：

* 空指针
* 数据库连接失败
* 参数解析异常
* 第三方服务超时

# 十五、环境知识：很多接口问题其实是环境问题

你必须分清：

* 开发环境
* 测试环境
* 预发布环境
* 生产环境

要知道：

* 每个环境的地址不一样
* 数据不一样
* 配置不一样
* 权限不一样

常见问题：

* 调错环境
* 用错账号
* 测试数据污染
* 环境服务没启动

# 十六、测试数据管理也很重要

接口测试经常依赖数据。

你要学会：

* 构造测试数据
* 维护账号数据
* 清理脏数据
* 隔离测试数据
* 准备前置数据

例如测下单：

* 要先有用户
* 要先有商品
* 要有库存
* 要有收货地址

所以很多接口测试失败，不是接口错，是前置数据没准备好。

# 十七、Mock 和依赖隔离

真实项目中，一个接口常常依赖别的系统。

例如：

* 支付接口依赖支付平台
* 发货接口依赖物流系统
* 登录接口依赖认证中心

当依赖系统不可用时，就会用 **Mock**。

你至少要知道：

* 什么是 Mock
* 为什么要 Mock
* 哪些场景适合 Mock
* Mock 返回值如何设计

# 二十、Bug 定位与提单能力

会测还不够，还要会说清楚问题。

一个合格的接口 Bug 至少写清楚：

* 接口名称
* 请求地址
* 请求方法
* 请求参数
* 请求头
* 实际结果
* 预期结果
* 环境
* 复现步骤
* 响应报文
* 日志或截图
* 影响范围

例如不是只写：

“接口报错了”

而是写成：

“测试环境，调用创建订单接口 `/api/order/create`，当 `productId` 为空时，接口返回 500。预期应返回 400 和明确的参数校验提示。”

# 二十三、很多新人最容易犯的错误

## 1. 只会点 Postman，不懂原理

工具只是表面，核心是 HTTP 和业务逻辑。

## 2. 只测正常流程

异常、边界、权限，才最容易发现问题。

## 3. 只看返回码，不查数据

返回成功不等于业务成功。

## 4. 不懂业务

接口测试本质上是测业务规则。

## 5. 不会定位问题

发现问题后说不清，是很吃亏的。
