## 1. 域名是什么，有什么特性

### 什么是域名

域名就是网站或服务在网络上的“名字”，用于替代难记的 IP 地址。

例如：

```text
api.example.com
```

它本质上会通过 DNS 解析到某个 IP，再去访问对应服务器。

### 为什么要有域名

因为 IP 不好记，而且服务部署可能会变。
域名不变，后端 IP 可以变，这样客户端不用跟着改。

### 域名的几个重要特性

#### 第一，具有层级结构

例如：

```text
api.test.company.com
```

从右往左看：

* `com`：顶级域
* `company`：主域
* `test`：二级或环境标识
* `api`：服务标识

很多公司会用子域名区分环境和服务：

* `api.dev.xxx.com`：开发环境
* `api.test.xxx.com`：测试环境
* `api.xxx.com`：正式环境

#### 第二，可以和具体服务解耦

域名指向的服务器可以换，但客户端访问方式不变。

#### 第三，可以做环境隔离

同一个系统常常会有不同域名对应不同环境，避免把测试请求打到生产。

#### 第四，可以配合 HTTPS 证书使用

HTTPS 证书通常和域名绑定，所以域名和安全通信密切相关。

### 接口测试里你要注意什么

你要确认：

* 域名是不是对的环境
* HTTP 还是 HTTPS
* 是否需要代理/VPN
* DNS 是否可解析
* 是否有跨域、证书、网关限制

## 2. 鉴权的相关概念

“鉴权”这个词，很多时候会和“认证”混着说，但严格来说它们不一样。

### 认证 Authentication

回答的是：

**你是谁？**

比如：

* 用户名密码登录
* 手机验证码登录
* Token
* JWT
* Session

### 授权 / 鉴权 Authorization

回答的是：

**你能做什么？**

比如：

* 普通用户只能看自己的订单
* 管理员可以删除用户
* 财务角色可以看账单，普通员工不能看

所以通常流程是：

**先认证身份，再判断权限。**

## 3. 路径参数不同代表什么

路径参数通常放在 URL 路径里，用来表示“资源身份”或者“资源层级”。

例如：

```text
GET /users/1001
GET /orders/20250001
GET /users/1001/orders/20250001
```

### 代表什么

#### 第一，资源的唯一标识

`/users/1001` 里的 `1001` 表示用户 ID。

#### 第二，资源之间的层级关系

`/users/1001/orders/20250001`
表示“用户 1001 的订单 20250001”。

#### 第三，资源类型不同

例如：

* `/products/1`
* `/categories/1`

这里的 `1` 都是 ID，但路径不同，含义不同。

### 为什么路径参数不同很重要

因为它往往决定你到底在访问谁的数据。

接口测试里要测：

* 正常 ID 是否能查到
* 不存在的 ID 会怎样
* 别人的 ID 能不能查到
* ID 类型错误会怎样
* 负数、0、超大值会怎样

---

## 4. 怎么理解查询参数

查询参数就是 URL 中 `?` 后面的部分，常用于“筛选、排序、分页、条件限定”。

例如：

```text
GET /orders?page=1&size=10&status=paid
```

### 它的作用

它不是在表示“哪个具体资源”，而是在描述“我要怎么查”。

### 常见用途

#### 分页

```text
?page=1&size=10
```

#### 条件过滤

```text
?status=paid
```

#### 排序

```text
?sort=createTime&order=desc
```

#### 搜索关键字

```text
?keyword=phone
```

### 路径参数和查询参数的区别

#### 路径参数

更像“你要找谁”

```text
/users/1001
```

#### 查询参数

更像“你怎么查”

```text
/users?name=Tom&age=20
```

## 5. 请求头和请求体中分别有什么

这两个非常重要。

---

### 请求头 Header 里通常放什么

Header 是“附加说明”和“控制信息”，不是业务主体数据。

常见内容有：

#### 1）内容格式

```text
Content-Type: application/json
```

表示请求体是什么格式。

#### 2）认证信息

```text
Authorization: Bearer xxxxx
```

表示身份凭证。

#### 3）客户端期望

```text
Accept: application/json
```

#### 4）Cookie / Session

```text
Cookie: JSESSIONID=xxx
```

#### 5）来源、设备、版本信息

例如：

* app 版本号
* 渠道号
* 设备类型
* traceId
* 请求链路标识

### 请求体 Body 里通常放什么

Body 放的是“业务数据本身”。

例如登录接口：

```json
{
  "username": "test01",
  "password": "123456"
}
```

例如创建订单：

```json
{
  "userId": 1001,
  "productId": 2001,
  "count": 2,
  "addressId": 3001
}
```

### 怎么理解二者区别

#### Header

像快递外面的标签：

* 发给谁
* 是什么格式
* 有没有授权
* 怎么处理

#### Body

像包裹里面真正的东西：

* 用户名密码
* 商品信息
* 订单数据

---

## 6. 校验 JSON 字段是什么意思

就是检查接口返回或请求中的 JSON 数据，是否“结构对、类型对、值对、规则对”。

例如返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "userId": 1001,
    "name": "Tom",
    "age": 18
  }
}
```

### 校验 JSON 字段一般校验什么

#### 1）字段是否存在

比如必须有 `code`、`message`、`data`。

#### 2）字段类型是否正确

* `userId` 应该是整数
* `name` 应该是字符串
* `data` 应该是对象
* `list` 应该是数组

#### 3）字段值是否正确

比如登录成功时：

* `code = 0`
* `message = success`

#### 4）字段是否为空

比如用户昵称允许为空吗，手机号能为空吗。

#### 5）字段格式是否正确

例如：

* 日期格式
* 手机号格式
* 邮箱格式

#### 6）字段之间的逻辑关系是否正确

例如：

* `totalAmount = itemAmount + freightAmount`
* `orderStatus=PAID` 时，`payTime` 不应为空

### 为什么要校验

因为接口可能“返回成功了，但内容不对”。

## 7. 表单格式、上传、XML 分别是什么

### 一，表单格式是什么

常见有两种。

#### 1）application/x-www-form-urlencoded

这是最传统的表单格式。

例如：

```text
username=test01&password=123456
```

特点：

* 键值对
* 常见于简单表单提交
* 不适合传文件

#### 2）multipart/form-data

用于文件上传，也可同时带普通字段。

例如上传头像时：

* 文件内容
* 用户 ID
* 备注说明

特点：

* 支持文件
* 一个请求里可以有多个部分
* 文件上传最常见

---

### 二，上传是什么

上传就是客户端把文件发给服务端。

常见场景：

* 上传头像
* 上传 Excel
* 上传图片
* 上传合同 PDF

测试时常关注：

* 文件大小限制
* 文件类型限制
* 文件名特殊字符
* 空文件
* 重复上传
* 上传后地址是否正确返回

---

### 三，XML 是什么

XML 是一种结构化数据格式，和 JSON 类似，但写法不同。

例如：

```xml
<user>
  <id>1001</id>
  <name>Tom</name>
</user>
```

### XML 的特点

* 标签化结构
* 可读性强
* 老系统、银行、支付、企业集成里比较常见
* 现在新系统更多用 JSON

### 接口测试里你要会什么

至少要看得懂：

* 根节点
* 子节点
* 属性
* 嵌套关系

---

## 8. 认证和鉴权详细讲解各种情况

这是接口测试里的重点。

---

### 一，认证：证明你是谁

### 常见认证方式

#### 1）用户名 + 密码

最基础。

流程：

1. 提交账号密码
2. 服务端校验
3. 成功后返回 session 或 token

测试要点：

* 正确账号密码
* 错误密码
* 账号不存在
* 密码为空
* 连续输错锁定
* 密码加密传输

---

#### 2）短信验证码 / 邮箱验证码

流程：

1. 获取验证码
2. 输入验证码登录

测试要点：

* 验证码是否过期
* 错误验证码
* 重复使用验证码
* 验证码次数限制
* 发送频率限制

---

#### 3）Session + Cookie

早期 Web 常见。

流程：

1. 登录成功
2. 服务端创建会话 Session
3. 返回 Session ID 给浏览器
4. 浏览器放在 Cookie 中
5. 后续请求自动带 Cookie

测试要点：

* 不带 Cookie 是否未登录
* Cookie 失效后是否需要重新登录
* 不同浏览器会话是否隔离
* 登出后 Session 是否失效

---

#### 4）Token

现在前后端分离最常见。

流程：

1. 登录成功
2. 服务端返回 token
3. 后续请求 Header 带 token

例如：

```text
Authorization: Bearer abcdefg
```

测试要点：

* 不带 token
* token 过期
* token 篡改
* token 和用户不匹配
* 登出后 token 是否失效

---

#### 5）JWT

JWT 是一种常见 token 格式。

它特点是：

* 自包含信息
* 服务端可不存会话
* 常见于分布式系统

测试要点：

* 是否过期
* 签名是否被验证
* payload 被改动后是否拒绝
* 是否泄露敏感信息

---

#### 6）OAuth 2.0

第三方授权常见，比如“微信登录”“GitHub 登录”。

它本质不是简单登录，而是“授权某个应用代表用户做有限操作”。

测试要点：

* code 是否只能用一次
* access token 是否过期
* refresh token 是否有效
* scope 权限范围是否正确

---

### 二，鉴权：判断你能做什么

鉴权主要分几类。

#### 1）按登录状态鉴权

例如：

* 未登录不能查订单
* 已登录可以查自己订单

#### 2）按角色鉴权

例如：

* 普通用户：只能查看自己的数据
* 客服：可以查看订单，但不能退款
* 管理员：可以删数据、改权限

#### 3）按资源归属鉴权

例如：

* 用户 A 只能访问用户 A 的地址信息
* 用户 B 不能访问 A 的订单详情

#### 4）按操作范围鉴权

例如：

* 可以读，不能写
* 可以新增，不能删除
* 可以导出，不能审批

---

### 三，接口测试里必须测的鉴权场景

#### 1）未登录访问

预期一般是 `401`

#### 2）已登录但权限不足

预期一般是 `403`

#### 3）水平越权

A 用户访问 B 用户资源

例如：

```text
GET /orders/10086
```

如果这个订单是别人账号的，是否还能查到。

#### 4）垂直越权

普通用户访问管理员接口

例如：

```text
POST /admin/user/delete
```

#### 5）伪造身份

替换 token、篡改 userId、篡改请求头

#### 6）前端限制但后端没限制

前端按钮灰掉了，但直接发接口还能成功，这就是后端鉴权漏洞。

### 一，异常场景

#### 用户名长度要求 6 到 20 位

要测：

* 5 位
* 6 位

* 21 位

#### 年龄要求 0 到 120

要测：

要测：

## 10. 状态流转中的非法状态是什么

状态流转是指一个业务对象在生命周期中状态如何变化。

例如订单状态：

```text
待支付 -> 已支付 -> 已发货 -> 已完成
待支付 -> 已取消
已支付 -> 退款中 -> 已退款
```

### 非法状态是什么意思

就是“不符合业务规则”的状态变化。

### 例子

#### 1）已取消订单还能支付

`已取消 -> 已支付`
通常这是非法状态。

#### 2）未支付订单直接发货

`待支付 -> 已发货`
通常非法。

#### 3）已完成订单再次取消

`已完成 -> 已取消`
通常非法。

#### 4）退款完成后还能再次退款

`已退款 -> 退款中`
通常非法。

### 为什么要测

因为状态管理一旦出错，业务就会乱。

## 11. 幂等性中的连续调用两次和同一请求重复提交有什么区别

这两个容易混，但不完全一样。

---

### 一，连续调用两次

指的是客户端真的发了两次请求，但不一定是“同一个请求实例”。

例如：

* 用户快速点了两次“提交订单”
* 脚本连续调用两次接口

它强调的是：

**同一个接口被多次调用，系统结果是否合理。**

例子：

* 查询接口调两次，结果应一致
* 删除接口调两次，第一次成功，第二次应合理返回“已删除”或“资源不存在”
* 支付接口调两次，不应该扣两次款

---

### 二，同一请求重复提交

更强调这是“同一业务请求”因为网络、重试、前端重复点击等原因被再次发送。

例如：

* 请求体完全一样
* requestId 完全一样
* 订单号完全一样
* 幂等号完全一样

它强调的是：

**同一业务动作不能被处理两次。**

例如支付：

* 第一次请求超时了，客户端没收到响应
* 客户端重试一次
* 服务端必须识别这是同一笔支付，不能再次扣款

---

### 区别总结

#### 连续调用两次

关注“接口多次被调用后，系统行为是否合理”。

#### 同一请求重复提交

关注“同一个业务动作被重复发送时，系统能否识别并避免重复处理”。

### 在接口测试中怎么测

你可以测：

* 同一接口快速发两次
* 完全相同请求发两次
* 同一个幂等号发两次
* 超时后重试
* 前端双击按钮

---

## 12. 详细讲解一下数据库和 Linux 日志

这部分我从“接口测试怎么用”来讲。

---

# A. 数据库

### 一，为什么接口测试要懂数据库

因为很多接口不能只看返回值，要看数据是否真的正确落库、更新、回滚。

比如：

* 注册成功后，用户表是否真的插入
* 修改密码后，数据库是否真的更新
* 下单后，订单表、库存表是否都变了
* 失败时，是否产生脏数据

---

### 二，测试人员最常用的数据库能力

#### 1）查询数据

最常用的是 `select`

例如查用户是否注册成功：

```sql
select * from user where phone='13800138000';
```

#### 2）验证更新结果

例如修改昵称后：

```sql
select nickname from user where user_id=1001;
```

#### 3）准备测试数据

例如插入一条商品数据、构造一条订单状态数据。

#### 4）清理测试数据

例如删除脏数据、回收测试账号。

---

### 三，接口测试里数据库重点关注什么

#### 1）数据是否新增成功

比如注册、创建订单、创建地址。

#### 2）数据是否更新正确

比如修改密码、修改状态。

#### 3）数据是否删除/逻辑删除正确

例如删除地址后，数据库是真删还是标记删除。

#### 4）多表是否一致

例如下单时可能涉及：

* 订单表
* 订单明细表
* 库存表
* 优惠券表

#### 5）失败时是否回滚

例如支付失败后：

* 订单状态不能错误更新
* 库存不能被错误扣减

---

### 四，常见数据库问题

#### 1）接口返回成功，但没入库

可能是异步、事务未提交、代码 bug。

#### 2）部分表更新成功，部分失败

说明事务有问题。

#### 3）重复数据

说明幂等没做好，或者唯一约束没做好。

#### 4）脏数据

比如订单失败了，却留下半条记录。

#### 5）并发问题

例如超卖、重复扣减库存、锁等待。

---

### 五，测试人员最该懂的数据库概念

#### 主键

唯一标识一条记录。

#### 唯一约束

防止重复数据，比如手机号不能重复注册。

#### 索引

提高查询速度。

#### 事务

多步数据库操作要么全成功，要么全失败。

#### 回滚

事务失败后恢复原状。

#### 锁

多个请求同时改同一数据时，会有锁竞争。

---

# B. Linux 日志

### 一，为什么要看日志

接口报 500、超时、失败时，最直接的证据往往在日志里。

日志可以帮你判断：

* 参数解析失败
* 空指针
* 数据库连接失败
* 第三方接口超时
* 权限不足
* 服务没启动
* 网关拦截

---

### 二，常见日志类型

#### 1）应用日志

最重要。
比如 Java 服务的业务日志、异常栈。

#### 2）Web 服务器日志

比如 Nginx 访问日志、错误日志。

#### 3）数据库日志

查询慢、连接失败、死锁等。

#### 4）容器日志

如果服务跑在 Docker/K8s 中，还会有容器日志。

---

### 三，测试常见怎么看日志

#### 看实时日志

```bash
tail -f app.log
```

#### 搜关键词

```bash
grep "ERROR" app.log
grep "orderId=1001" app.log
```

#### 看最近几百行

```bash
tail -n 200 app.log
```

#### 看网关日志

确认请求有没有到达服务端。

---

### 四，接口测试里如何结合日志定位问题

#### 场景 1：接口返回 500

先看：

* 应用异常栈
* 空指针
* JSON 解析异常
* SQL 执行异常

#### 场景 2：接口超时

看：

* 服务是否卡住
* 下游依赖是否超时
* 数据库是否慢查询
* 网络是否异常

#### 场景 3：明明传了 token 还说未登录

看：

* 网关是否转发了 Header
* token 解析是否失败
* token 是否过期

#### 场景 4：接口返回成功但数据不对

看：

* 业务日志里状态是否走错
* SQL 更新了哪张表
* 是否命中了缓存
* 是否异步处理未完成

---

## 13. 测试数据管理常见的问题和处理方法

测试数据管理是很多新手最容易忽略、但实际工作特别重要的部分。

### 一，常见问题

#### 1）数据被别人改了

多人共用测试环境时，经常出现：

* 你刚准备好的订单被别人支付了
* 你的商品被别人下架了

#### 2）测试数据不完整

例如测下单时缺：

* 用户
* 商品
* 库存
* 地址
* 优惠券

#### 3）数据重复

例如：

* 手机号重复注册
* 重复订单号
* 文件名重复

#### 4）脏数据越来越多

接口失败后留下无效记录，影响后续测试。

#### 5）环境数据和预期不一致

文档说商品有库存，实际上数据库里库存为 0。

#### 6）依赖链太长

测一个接口要先准备 5 个前置接口，任何一步失败都没法继续。

#### 7）测试数据不可追溯

过几天不知道这条数据是谁造的、什么时候造的、用来干什么。

---

### 二，处理方法

#### 1）建立固定测试账号

例如：

* 普通用户账号
* 管理员账号
* 冻结账号
* 黑名单账号

#### 2）测试数据分层

把数据分为：

* 公共基础数据
* 个人临时数据
* 自动化专用数据

#### 3）用脚本初始化数据

不要每次手动造数据，尽量脚本化。

#### 4）每次测试后清理

例如删掉测试订单、回滚库存、清理上传文件。

#### 5）使用唯一标识

例如手机号、用户名、订单备注加时间戳，避免冲突。

#### 6）环境隔离

重要测试尽量别和别人共用同一批核心数据。

#### 7）记录数据字典

说明：

* 哪些账号做什么用
* 哪些商品不能乱动
* 哪些数据每天会重置

---

## 14. 实际工作中的整个业务流程测试步骤和常见问题

我用一个比较典型的例子：**电商下单流程**。

业务流程：

```text
登录 -> 查询商品 -> 加入购物车 -> 提交订单 -> 支付 -> 查询订单 -> 取消/退款
```

---

### 第一步：看需求和接口文档

先弄清楚：

* 有哪些接口
* 每个接口做什么
* 参数规则是什么
* 状态流转是什么
* 权限限制是什么

常见问题：

* 文档不完整
* 字段说明不一致
* 返回码没有定义
* 状态规则说不清

---

### 第二步：梳理业务流程和依赖关系

例如下单依赖：

* 用户已登录
* 商品存在且上架
* 库存足够
* 地址存在
* 优惠券有效

常见问题：

* 只盯单个接口，忽略上下游关系
* 不知道哪个数据是前置条件

---

### 第三步：设计测试用例

要覆盖：

#### 正常流程

* 登录成功
* 下单成功
* 支付成功
* 查询成功

#### 异常流程

* 未登录下单
* 商品下架
* 库存不足
* 地址不存在
* 支付失败
* 重复支付

#### 边界场景

* 数量 = 1
* 数量 = 最大购买数
* 金额 = 0
* 优惠券恰好过期边界

#### 鉴权场景

* A 用户查 B 用户订单
* 普通用户调用后台接口

常见问题：

* 只测主流程
* 不测状态流转
* 不测越权
* 不测幂等

---

### 第四步：准备环境和测试数据

包括：

* 登录账号
* 商品数据
* 库存
* 地址
* 支付方式
* 测试用优惠券

常见问题：

* 环境错了
* token 过期
* 商品没库存
* 地址不是当前用户的
* 优惠券已失效

---

### 第五步：先做接口冒烟

先验证接口最基本可用：

* 能不能通
* 参数格式对不对
* 是否需要鉴权
* 返回结构是否正常

常见问题：

* 域名错误
* 网关未放通
* Header 缺失
* Content-Type 错误

---

### 第六步：按业务链路联调

示例：

#### 1）登录

拿到 token

#### 2）查询商品

确认商品状态和库存

#### 3）提交订单

检查返回订单号

#### 4）支付订单

检查支付结果

#### 5）查订单详情

核对状态是否从：
`待支付 -> 已支付`

#### 6）查数据库

确认：

* 订单表正确
* 明细表正确
* 库存扣减正确

#### 7）必要时看日志

确认：

* 调用了哪个下游服务
* 是否有异常栈
* 是否有超时

常见问题：

* token 传错
* 请求链某一步没做
* 状态没更新
* 数据库没落对
* 异步任务延迟导致“查的时候还没更新”

---

### 第七步：验证异常和恢复能力

例如：

* 支付接口超时后重试
* 重复提交订单
* 库存同时被多个请求扣减
* 第三方支付失败
* 下游服务不可用

常见问题：

* 状态卡住
* 部分成功、部分失败
* 重复扣款
* 库存没回滚
* 出现脏数据

---

### 第八步：回归和提 Bug

提 Bug 时写清楚：

* 哪个接口
* 请求参数
* 环境
* token/账号
* 实际结果
* 预期结果
* 日志证据
* 数据库证据

常见问题：

* Bug 描述太模糊
* 没有复现步骤
* 没有原始请求报文
* 没有日志/查库证据

---

# 最后给你一个工作中最实用的测试顺序

你真的开始做一个新业务接口测试时，可以照这个顺序：

1. 看需求和文档
2. 画业务流程
3. 找出前置条件和依赖接口
4. 准备测试数据和账号
5. 先跑通主流程
6. 再测异常、边界、等价类
7. 再测鉴权、状态流转、幂等
8. 查数据库确认结果
9. 看日志定位问题
10. 清理测试数据并做回归

---

# 一句话总结

接口测试真正做起来，不是“发个请求看返回”，而是要同时看：

**协议、参数、权限、状态、数据落库、日志、上下游依赖、异常恢复和测试数据。**
