# 检索准确性专项报告

## 1. 测试范围

- 测试集：`tests/metadata_and_retrieval_cases.json` 中 `retrieval` 类用例，共 `25` 条
- 检查项：
  - `Top1` 是否命中目标文档
  - `Top3` 是否包含目标文档
  - `Top3` 是否排除禁用来源
  - 检索到的 `chunks` 是否包含预期关键词

说明：本报告聚焦“检索层准确性”，不把答案生成是否完整纳入通过标准。

## 2. 基线结果

在本轮修改前，基于更新后的检索链路做 `sources` 专项评测，结果为：

- 检索专项通过率：`23 / 25 = 92.0%`
- 剩余失败用例：`R03`、`R15`

失败明细：

1. `R03` `开路/短路检测的判定标准是什么？`
   - 现象：`Top1` 命中 `测试/测试用例/开路短路测试用例集.md`
   - 期望：`测试/测试项/晶圆CP测试项定义与规范.md`
   - 原因：问题里有“判定标准”，但系统更偏向“测试用例”而不是“测试规范/测试项”

2. `R15` `WPS-3100 机型的核心技术指标和温控范围是多少？`
   - 现象：`Top1` 正确，但 `Top3` 中仍混入 `WPS-3000` / `WPS-3200`
   - 原因：同一产品族机型相似度高，`WPS-3100` 的精确型号约束不够强

## 3. 根因分析与针对性修改

### 3.1 测试规范 vs 测试用例混淆

问题类型：

- 关键词如 `判定标准`、`定义与规范`、`测试项`
- 实际应该优先检索 `test_spec`
- 旧逻辑没有显式区分 `test_spec` / `test_case`

修改：

- [core/query_understanding.py](/Users/raoziyu/Documents/code/VideoCode-main/使用Python构建RAG系统/rag/core/query_understanding.py)
  - 新增 `test_spec` / `test_case` 意图
  - 对 `判定标准`、`测试规范`、`定义与规范` 做高优先级识别
- [core/store.py](/Users/raoziyu/Documents/code/VideoCode-main/使用Python构建RAG系统/rag/core/store.py)
  - 新增 `query -> doc_type` 显式加权
  - 问“判定标准/规范”时提升 `test_spec`，压低 `test_case`
- [core/rag_chain.py](/Users/raoziyu/Documents/code/VideoCode-main/使用Python构建RAG系统/rag/core/rag_chain.py)
  - 在 rerank 阶段追加同样的 `doc_type` 语义加权

### 3.2 同产品族型号混淆

问题类型：

- `WPS-3100` 查询会混入 `WPS-3000`、`WPS-3200`
- 旧逻辑对“同前缀但不同型号”只有弱约束

修改：

- [core/store.py](/Users/raoziyu/Documents/code/VideoCode-main/使用Python构建RAG系统/rag/core/store.py)
  - 新增 `_model_match_adjustment`
  - 精确型号命中加分
  - 同产品族但错型号时显式降权
- [core/rag_chain.py](/Users/raoziyu/Documents/code/VideoCode-main/使用Python构建RAG系统/rag/core/rag_chain.py)
  - 在 rerank 分数中追加型号命中加分/错型号降权

### 3.3 之前已完成并继续保留的修正

- `PCK-1002` 类告警码不再误识别为机型，转为 `fault_code/work_order`

## 6. 残余问题

当前主要剩余问题已不在检索层，而在答案生成层：

- 有些问题文档和 chunk 都找对了，但回答没有完整引用关键数值或编号
- 全量 API 评测时，长时间连续 SSE 请求偶发连接异常，影响全量跑批稳定性

下一步建议：

1. 优化 `generate_answer()` 提示词，让回答更强制抽取数字、编号、阈值
2. 给评测脚本增加“仅检索模式”，避免为了测检索还必须等待整段答案生成
3. 给 SSE 测试脚本加入重试和连接异常容错
