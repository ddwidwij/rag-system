# 信息使能AI平台 — 基于现有代码基础的详细可执行需求文档

> **基准版本**：当前 RAG 系统已实现混合检索（BM25 + Vector + Rerank）、元数据多字段过滤、Markdown 结构感知分块、按 source 去重、版本号归一化、审计日志、文档规则检查器（checker.py）。以下需求均在此基础上迭代，按优先级排序，每项包含技术实现路径和验收标准。

---

## 模块一：检索质量优化

### FR-1.1 查询扩展与同义词处理

**背景**：当前查询直接送入向量检索，对"排障"/"故障排查"/"troubleshoot"等同义表达召回率低。

**需求描述**：
- 在 `retrieve_hybrid` 调用前，新增 `expand_query(query: str) -> list[str]` 函数
- 扩展策略：LLM 重写 + 同义词词典两路并行，合并后去重
- 同义词词典以 JSON 文件 `core/synonyms.json` 维护，格式：`{"故障": ["报警", "异常", "错误", "error"], ...}`
- 多路 query 的向量检索结果做 RRF 融合后再进入 rerank

**实现位置**：store.py 新增 `expand_query`；server.py 的 `_stream_rag` 在调用 `retrieve_hybrid` 前调用

**验收标准**：对测试集中同义词问题的命中率从基准提升 ≥ 15%；扩展耗时 ≤ 500ms

---

### FR-1.2 BM25 中文分词优化

**背景**：当前 `retrieve_hybrid` 用空格分词，中文 BM25 效果差。

**需求描述**：
- 在 store.py 的 BM25 构建段，引入 `jieba` 分词替代空格切词
- 对 query 和 corpus 文本均做 jieba 分词
- 支持加载自定义词典 `core/user_dict.txt`（格式同 jieba）

**实现位置**：store.py 的 `retrieve_hybrid` 函数内

**验收标准**：BM25 对中文专业术语（如"运动控制模块故障"）的召回精度可用人工抽检验证，issue 数 ≤ 2/10 条测试

---

### FR-1.3 低置信度兜底策略

**背景**：当 `max_rerank_score < RERANK_LOW_CONFIDENCE_THRESHOLD` 时，当前只返回提示，未做兜底。

**需求描述**：
- 低置信度时，触发"宽松召回"：放宽 `where` 条件（去掉 version/module 过滤，仅保留 product_line），重新检索一次
- 若宽松召回仍无结果，返回相关度最高的 Top-3 chunk 并附"参考内容，非精确匹配"标注
- 在 `sources` SSE 事件中增加 `fallback: true` 字段供前端区分展示

**实现位置**：server.py 的 `_stream_rag` 函数，`RERANK_LOW_CONFIDENCE_THRESHOLD` 可在 config.py 调整

**验收标准**："无结果"比例从当前基准下降 ≥ 30%；兜底结果的用户满意率（反馈按钮）≥ 50%

---

### FR-1.4 知识缺口自动识别

**背景**：audit.log 已记录每次查询及 `low_confidence` 字段，但未做分析。

**需求描述**：
- 新增 `tools/gap_analysis.py` 脚本，读取 audit.log（NDJSON 格式）
- 统计维度：low_confidence 比例、高频零命中 query、命中集中度（哪些文档被高频引用，哪些从未被引用）
- 输出 HTML 报告 `reports/gap_YYYY-MM-DD.html`，包含：Top-20 零命中问题、Top-20 高频命中来源、从未被引用文档列表
- 支持 CLI：`python -m tools.gap_analysis --days 7`

**实现位置**：新建 `tools/gap_analysis.py`

**验收标准**：可在 CI 中每日自动运行，报告可正常生成；Top-20 零命中 query 与人工观察吻合率 ≥ 80%

## 模块二：文档解析增强

### FR-2.1 PDF/DOCX 章节级解析（Track 2 升级）

**背景**：当前 parsers.py 中 `.pdf` 和 `.docx` 走 `rich_text` 轨道，但实现为 stub（返回空列表或简单段落切分），无章节感知。

**需求描述**：
- `.docx`：使用 `python-docx` 解析，按 Heading 1/2/3 做章节级切块，表格整体保留（同 Markdown 表格处理逻辑）
- `.pdf`：引入 `pdfminer.six` 或 `pymupdf`，提取文本并按"大间距/字号变化"推断章节边界
- 切块大小上限：2000 字符，超长章节滑窗 overlap=1 段
- 自动抽取文档标题（`.docx` 取 `core_properties.title`，`.pdf` 取首行大字号文本）补充至 `metadata["title"]`

**实现位置**：parsers.py 的 `_split_docx_chunks`、`_split_pdf_chunks` 函数

**验收标准**：对 10 份真实技术文档，章节切块正确率 ≥ 90%，表格不被截断率 100%

---

### FR-2.2 表格结构化抽取（Track 3 升级）

**背景**：`.xlsx/.csv` 当前逐行转文本，丢失列名语义。

**需求描述**：
- 每行转为"列名: 值"格式的自然语言片段，如 `参数名: max_speed | 默认值: 100 | 单位: rpm | 适用版本: V3.2`
- 保留表头作为每个 chunk 的前缀
- 支持多 Sheet：每个 Sheet 作为独立文档段落，前缀加 `[Sheet: {名称}]`
- 空行自动跳过；超长单元格截断至 500 字符

**实现位置**：parsers.py 的 `_split_table_chunks` 函数

**验收标准**：对参数规格表，检索"max_speed 默认值"可精确命中对应行，准确率 ≥ 95%

---

### FR-2.3 DITA 语义增强解析

**背景**：当前 DITA 解析只做 XML 文本提取，未利用 DITA 语义标签。

**需求描述**：
- 识别 DITA topic 类型（`concept`/`task`/`reference`）并写入 `metadata["doc_type"]`（若未被 meta.json 覆盖）
- `<step>/<cmd>/<info>` 块整体保留为一个 chunk，不跨步骤切分
- `<table>` 整体保留（同 Markdown 表格）
- 抽取 `<shortdesc>` 作为 chunk 摘要，写入 `metadata["summary"]` 字段（需在 `META_FIELDS` 和 `DEFAULT_META` 中增加该字段）
- `<note type="warning/caution/danger">` 整体保留，前缀加 `[WARNING]`/`[CAUTION]`/`[DANGER]`

**实现位置**：parsers.py 的 `_split_dita_chunks`；config.py 增加 `summary` 字段

**验收标准**：DITA 步骤类文档的操作步骤不被切断，step chunk 完整率 100%

## 模块三：元数据与知识治理

### FR-3.1 元数据字段扩展

**背景**：当前 `META_FIELDS` 缺少需求文档 3.3 节中的多个业务关键字段。

**需求描述**：在 config.py 增加以下字段（同步更新 `DEFAULT_META`、`QueryRequest`、`/api/ingest`、`/api/meta-options`）：

| 新增字段 | 说明 | 示例值 |
|---|---|---|
| `model_type` | 机型（已有，确认路由正常） | WPS-3000 |
| `project_id` | 关联项目编号 | PRJ-2025-001 |
| `lifecycle_state` | 生命周期状态 | 新建/审核中/生效/归档/下线 |
| `related_alarm_code` | 关联报警码 | E-401 |
| `related_test_item` | 关联测试项 | 运动控制-速度精度测试 |
| `related_hardware_version` | 关联硬件版本 | HW-V1.2 |
| `summary` | 文档/chunk 摘要（自动抽取或手填） | 本文描述... |

**实现位置**：config.py，并级联更新 server.py 中的 `QueryRequest`、`_build_where`、`/api/ingest`

**验收标准**：新字段可在 `/api/ingest` 中写入，`/api/meta-options` 可返回对应下拉选项，`/api/query` 可按新字段过滤

---

### FR-3.2 文档版本冲突检测

**背景**：同一文档存在多个版本时，RAG 可能混合返回旧版和新版内容，造成歧义。

**需求描述**：
- 在 `_stream_rag` 中，当 `reranked_evidence` 中出现同一 `doc_id`（非空）的多个不同 `version` 时，在 `sources` SSE 事件中增加 `version_conflict: [{doc_id, versions: [...]}]` 字段
- 前端在证据卡片区域显示"⚠️ 存在版本冲突，已返回多版本内容"提示
- 新增配置 `PREFER_LATEST_VERSION = True`（config.py）：开启时自动按 `effective_date` 降序，仅保留最新版本的 chunk

**实现位置**：server.py 的 `_stream_rag`；config.py 增加开关；index.html 前端提示

**验收标准**：构造含两个版本同文档的测试集，冲突检测准确率 100%；开启 PREFER_LATEST_VERSION 后旧版 chunk 不出现在结果中

---

### FR-3.3 文档有效性治理接口

**背景**：审计日志已记录查询，但无文档老化监控。

**需求描述**：
- 新增 `GET /api/doc-health` 接口，返回：
  - `expired_docs`：`effective_date` 超过 `EXPIRE_AFTER_DAYS`（默认 365）的文档列表
  - `no_owner_docs`：`owner` 为空的文档列表
  - `draft_docs`：`status == "草稿"` 的文档列表
  - `never_cited_docs`：在 audit.log 中从未出现在 `evidence_sources` 的文档
- CLI 入口：`python -m tools.gap_analysis --health`

**实现位置**：server.py 新增路由；`tools/gap_analysis.py` 复用

**验收标准**：接口返回数据格式正确；对测试数据集中 5 份过期文档，全部出现在 `expired_docs` 中

## 模块四：文档质量检查增强

### FR-4.1 checker.py 扩展规则集

**背景**：checker.py 已实现基础规则，但覆盖面不足目标 1 的 80% 要求。

**需求描述**：新增以下检查规则：

| 规则 ID | 描述 | 级别 | 实现要点 |
|---|---|---|---|
| R-101 | 单位格式不规范（如 `100ms` 应写 `100 ms`） | warning | 正则匹配数字+单位无空格 |
| R-102 | 版本号格式不一致（同文档中 `v1.0`/`V1.0`/`1.0` 混用） | warning | 收集全文版本号格式集合 |
| R-103 | 表格列数不一致（同表不同行列数不同） | error | 按 `|` 分割计列数 |
| R-104 | 有序列表缩进错误 | warning | 检查子列表缩进是否为 2 或 4 空格 |
| R-105 | 图片引用路径不存在 | error | `![...](path)` 提取 path，检查文件存在 |
| R-106 | 代码块未标注语言 | info | ` ``` ` 后无语言标识 |
| R-107 | 句子以"等等"结尾（语气模糊） | info | 正则检测 |
| R-108 | 同一段落中出现互相矛盾的数值（如"最小值 100，最大值 50"） | warning | 提取数值对，LLM 辅助判断（可选） |

**实现位置**：`tools/checker.py`，在现有 `_RULES` 列表中追加

**验收标准**：对 20 份真实文档，新规则误报率 ≤ 10%；漏报率 ≤ 20%

---

### FR-4.2 DITA 结构完整性检查

**背景**：`checker.py` 目前只检查 Markdown，DITA 文档未覆盖。

**需求描述**：
- 新增 `tools/dita_checker.py`，使用 `xml.etree.ElementTree` 解析 `.dita`/`.ditamap`
- 检查项：
  - `<shortdesc>` 是否存在
  - `<title>` 是否为空
  - `<step>` 是否包含 `<cmd>`
  - `conref`/`keyref` 引用的目标文件是否存在
  - `<xref href>` 引用的锚点是否在 ditamap 中注册
  - `<note>` 的 `type` 属性是否为合法值（note/tip/warning/caution/danger）
- 与 checker.py 统一输出格式（`CheckIssue` dataclass），支持 JSON 输出和 HTML 报告

**实现位置**：新建 `tools/dita_checker.py`；server.py 的 `/api/check` 接口同时支持 `.md` 和 `.dita`

**验收标准**：对 10 份内部 DITA 文档，结构检查问题发现率 ≥ 80%（与人工评审对比）

---

### FR-4.3 批量检查 Web 接口

**背景**：`/api/check` 当前只支持单文件检查，无批量能力。

**需求描述**：
- 新增 `POST /api/check-batch`：接收多文件上传，返回每个文件的检查结果列表
- 新增 `GET /api/check-report?format=html|json`：对 `DEFAULT_DOCS_DIR` 下所有文档运行检查，生成汇总报告
- 报告包含：文件级问题汇总、规则级分布饼图（HTML 版）、问题总数/error 数/warning 数

**实现位置**：server.py 新增路由

**验收标准**：对 241 个文档批量检查，10 秒内完成（单进程）；报告格式验收通过

**背景**：用户无法看到历史查询，也无法对答案质量反馈。

**实现位置**：index.html；server.py 新增 `/api/feedback` 路由
